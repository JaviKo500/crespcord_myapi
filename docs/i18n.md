# Internationalization (i18n) of system messages

All system messages (errors and optional success messages) are translated
per request based on the `Accept-Language` header. Only **`es`** and **`en`**
are supported; the default is **`es`**.

This applies only to **system messages**. Response `data` (role names, user
data, etc.) is **not** translated.

## Language resolution — `Accept-Language`

| Header | Value |
|--------|-------|
| Accept-Language | `es` / `en` (optional) |

- The language is taken from the **first two characters** of `Accept-Language`,
  lowercased. Example: `es-ES,es;q=0.9,en;q=0.8` → `es`.
- Quality factors (`q`) are **not** negotiated — only the first 2-letter subtag
  is read.
- The value is validated against the whitelist `['es', 'en']`. If the header is
  missing or the language is unsupported (e.g. `fr`), it falls back to **`es`**.
- Resolved once per request (cached in a `static`).

There is **no** `?lang=` query parameter and the preference is **not** persisted
in the database or the token — it is resolved per request from the header only.

## Error envelope

```json
{
  "success": false,
  "error_code": "invalid_credentials",
  "error": "Usuario o contraseña incorrectos."
}
```

- **`error_code`** — stable catalogue key (English, snake_case), independent of
  the language. Use it for client logic.
- **`error`** — message translated into the resolved language, with any
  placeholders already substituted.

## Success envelope (optional `message`)

```json
// Without message_key (e.g. login): unchanged
{ "success": true, "data": { } }

// With message_key (e.g. a future logout):
{ "success": true, "data": { }, "message": "Sesión cerrada correctamente." }
```

The `message` field is only present when a resource passes a `message_key` to
`myapi_respond($data, $status, $message_key, $replacements)`.

If a key does not exist in the catalogue, `myapi_t()` returns the key itself as
a safe fallback (the response never breaks, and missing keys are easy to spot).

## Helpers

Defined in `includes/myapi.i18n.inc`:

- `myapi_get_lang()` — resolves and caches the request language (`es`/`en`).
- `myapi_t($key, $replacements = [], $lang = NULL)` — translates a key; defaults
  to `myapi_get_lang()` when `$lang` is `NULL`.

Resources never hard-code message text: they pass a **key** to `myapi_error()`
or `myapi_respond()`.

## Examples

Spanish (explicit):
```bash
curl -i -X POST https://host/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -H 'Accept-Language: es' \
  -d '{"username":"javier","password":"wrong"}'
# 401 → { "success": false, "error_code": "invalid_credentials",
#         "error": "Usuario o contraseña incorrectos." }
```

English:
```bash
curl -i -X POST https://host/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -H 'Accept-Language: en' \
  -d '{"username":"javier","password":"wrong"}'
# 401 → { "success": false, "error_code": "invalid_credentials",
#         "error": "Invalid username or password." }
```

No header (falls back to `es`):
```bash
curl -i -X POST https://host/api/v1/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"password":"x"}'
# 422 → { "success": false, "error_code": "missing_field",
#         "error": "Falta el campo requerido: username" }
```
